## Project Overview

**payload-next-starter** is a full-stack boilerplate combining:
- **Payload CMS 3** (admin UI, collections, globals, REST + local API)
- **Next.js 16** App Router (frontend + API routes in one process)
- **PostgreSQL** via `@payloadcms/db-postgres`
- **Tailwind CSS v4** with CSS variable design tokens
- **Resend** for transactional email and newsletter

The Payload admin and Next.js frontend share a single process. Payload's REST API is served by the catch-all route at `src/app/(payload)/api/[...slug]/route.ts`. Custom API routes live alongside it in `src/app/(payload)/api/<route-name>/route.ts`.

## Critical Rules

### Do not touch
- `src/app/(payload)/admin/` ‚Äî auto-generated by Payload, do not edit
- `src/payload-types.ts` ‚Äî auto-generated by `payload generate:types`, do not edit manually
- `src/app/(payload)/importMap.js` ‚Äî auto-generated

### Schema changes require migration
Any change to a Payload collection or global field definition requires a database migration. This project has `push: false` set in the postgres adapter ‚Äî Payload will **not** auto-push schema changes.

To generate and run a migration after changing a collection:
```bash
npx payload migrate:create
npx payload migrate
```

### Route placement for custom API endpoints
Custom API routes must go inside the `(payload)` route group to avoid conflicts with Payload's catch-all:
```
src/app/(payload)/api/<my-route>/route.ts   ‚úì correct
src/app/api/<my-route>/route.ts             ‚úó may conflict with Payload catch-all
```

### Use Payload local API in server-side code
In API routes, server components, and hooks, use the local API ‚Äî not fetch:
```ts
import { getPayload } from 'payload'
import config from '@payload-config'

const payload = await getPayload({ config })
const result = await payload.find({ collection: 'posts', ... })
```

## i18n

- All public pages live under `src/app/(frontend)/[locale]/`. The `locale` param is always `en` or `id`.
- UI strings are in `src/i18n/dictionaries/en.ts` (source of truth) and `src/i18n/dictionaries/id.ts`.
- The `Dictionary` type is inferred from `en.ts` ‚Äî `id.ts` must satisfy it exactly.
- When adding new UI strings, always update **both** dictionaries.
- Payload field localization: add `localized: true` to fields that need per-language values in the database.

## Email System

Email logic lives entirely in `src/lib/email/`:

| File | Purpose |
|------|---------|
| `client.ts` | Resend singleton, `EMAIL_FROM`, `ADMIN_EMAIL` constants, dev-mode flag |
| `templates.ts` | Pure functions returning `{ subject, html, text }` ‚Äî no side effects |
| `index.ts` | `sendContactNotification`, `sendContactAutoReply`, `addNewsletterSubscriber` |

**Dev mode behavior:** when `RESEND_API_KEY` is not set, email functions log to console instead of calling Resend. Never throw in dev mode due to a missing key.

**Adding a new email type:**
1. Add a template function to `templates.ts`
2. Add a sender function to `index.ts`
3. Call it from the relevant API route or Payload hook

**Required env vars:** `RESEND_API_KEY`, `EMAIL_FROM`, `ADMIN_EMAIL`
**Optional:** `RESEND_AUDIENCE_ID` (newsletter), `NEXT_PUBLIC_SITE_NAME`

## Coding Conventions

- **TypeScript** ‚Äî strict mode, no `any`. Use Payload's generated types from `payload-types.ts`.
- **Tailwind** ‚Äî use CSS variables (`var(--primary)`, `var(--foreground)`) for brand colors instead of hardcoded values. Design tokens are defined in `src/app/globals.css`.
- **Client components** ‚Äî add `'use client'` only when hooks or browser APIs are needed. Prefer server components.
- **Access control** ‚Äî use the helpers in `src/access/`. Do not inline access logic in collection configs.
- **No inline secrets** ‚Äî all credentials via environment variables only.

## Common Tasks

### Add a new collection
1. Create `src/collections/MyCollection.ts` with a `CollectionConfig`
2. Import and add to `collections: []` in `src/payload.config.ts`
3. Run `npx payload migrate:create && npx payload migrate`
4. Regenerate types: `npx payload generate:types`

### Add a new page route
1. Create `src/app/(frontend)/[locale]/my-page/page.tsx`
2. Call `getDictionary(locale)` for UI strings
3. Add the route to both i18n dictionaries and the sitemap if needed

### Add a new content block
1. Create `src/blocks/MyBlock.ts` (Payload block definition)
2. Add it to the `layout` field blocks array in `src/collections/Pages.ts`
3. Add a React renderer component in `src/components/blocks/MyBlock.tsx`

### Add a new email
1. Add template to `src/lib/email/templates.ts`
2. Add sender function to `src/lib/email/index.ts`
3. Call from an API route (`src/app/(payload)/api/`) or a Payload hook

### Change the sending domain
Update `EMAIL_FROM` in `.env`. The domain must be verified in your Resend dashboard with SPF/DKIM records.
